Data Display Skill

Design tables, lists, and review pages with structure first.

This skill separates structure, color intake, and token styling so both people and agents can build more reliable admin lists, operations pages, and resource dashboards.

Workflow

01

Theme Intake

Confirm color mood, palette family, and fixed hex values first. Do not lock the final visual direction before this step.

Structure Rules

03

Token Layer

Use the visual token layer to define color, surface, border, radius, shadow, and state styling without breaking structural priorities.

04

Final Page

Render the final data page and validate components, state modules, and responsive behavior with the preview page.

Theme Color Intake

Confirm the color direction before any visual implementation. Do not default to the sample blue token set.

Ask In This Order

1. Color mood
2. Palette family
3. Fixed hex values

Input Priority

1. User-provided hex values
2. Project `DESIGN.md` or brand VI
3. User-described mood and palette
4. Default token set

See `theme-color-intake.md` for the standard prompt. If the user gives only a mood such as calm or tech, propose 1-2 candidate directions before locking the final visual treatment.

Rule Layers

Structure is stronger than styling. Brand VI may override tokens, but must not break hierarchy, state clarity, or table readability.

Higher Priority

Structure First

`Table_List_System.md` controls page anatomy, search/filter structure, table column logic, state modules, and responsive behavior.

Lower Priority

Token Second

`Data_Display_DESIGN.md` and any project-level `DESIGN.md` handle color, radius, shadow, type, and branding only. They do not override structural priority.

Prompt Snippet

Read the entry docs first, then intake, then structure, then tokens, and only then generate the page. 

Read README.md and index.html first. Confirm the theme color direction with theme-color-intake.md. Then use Table_List_System.md as the structural source of truth and Data_Display_DESIGN.md as the token layer. If the project also has DESIGN.md, use it only to override tokens without breaking readability.
